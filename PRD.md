## 一、项目背景与目标

### 1.1 背景

客服团队（以租号客服为主）日常面临以下排班痛点：

- 每月手工收集轮休意愿，效率低、易出错；
- 节假日安排、调休工作日等特殊日期处理复杂；
- 早晚班分配不均、连班过长等合规性难以自动校验；
- 班次对调申请需要人工协调，沟通成本高。

### 1.2 目标

构建一套**全流程线上化排班管理系统**，实现：

1. **员工自助**预填下月轮休/请假意愿（日历交互）；
2. **管理员一键计算**排班，系统自动输出满足约束条件的方案；
3. **多方案管理**：保存、预览、应用、覆盖历史排班方案；
4. **班次对调**：员工发起申请 → 对方确认/拒绝 → 系统记录全程；
5. **调整日志**：所有排班变更、方案应用均有审计追踪；
6. **节假日 & 最低人数**配置化，无需修改代码即可适配每年假期。

## 四、数据库模型

数据库使用 **SQLite + Prisma**，主要模型如下：

### 4.1 `AuditLog`（审计日志 / 配置存储）

> 当前系统用 AuditLog 作为配置持久化的主要载体，通过 `action` 字段区分不同配置类型。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String (cuid) | 主键 |
| `actorId` | String | 操作人 ID |
| `action` | String | 配置 key（如 `holiday-config`、`min-staff-config`、`MEMBER_HOLIDAY_OVERRIDE:yyyy-MM`） |
| `details` | String | JSON 序列化的配置内容 |
| `createdAt` | DateTime | 创建时间 |

**常见 action 值：**

| action 值 | 对应配置 |
|---|---|
| `holiday-config` | 全局节假日配置数组 |
| `min-staff-config` | 每日类型最低排班人数 |
| `MEMBER_HOLIDAY_OVERRIDE:yyyy-MM` | 指定月份各成员法定节假日排班天数 |

### 4.2 其他模型（预留，当前未完全接入前端）

| 模型 | 说明 |
|---|---|
| `User` | 用户信息，含飞书 ID、角色（ADMIN/MANAGER/EMPLOYEE） |
| `Schedule` | 具体排班记录，含日期、班次、状态（DRAFT/PUBLISHED） |
| `WishDay` | 员工意愿休假日期 |
| `LeaveDay` | 请假记录 |
| `MonthlyRule` | 每月应排班天数规则（支持管理员覆盖） |
| `Holiday` | 节假日与调休工作日 |
| `Setting` | 全局设置（工作日/周末最低人数，连班上限） |

## 五、API 接口说明

### 5.1 节假日配置

#### `GET /api/config/holidays`

获取全局节假日配置列表。优先从数据库最新 `AuditLog` 读取，无记录时返回默认内置数据（覆盖 2026 年主要节假日）。

**响应示例：**
```json
[
  { "date": "2026/04/05", "name": "清明节", "type": "节假日", "isLegal": true },
  { "date": "2026/04/07", "name": "清明节", "type": "调休上班日", "isLegal": false }
]
```

#### `PUT /api/config/holidays`

保存节假日配置（覆盖写入），前端在「全局配置 → 节假日配置」点击「保存」时调用。

**请求体：** `HolidayRow[]`（同上格式数组）

---

### 5.2 最低排班人数配置

#### `GET /api/config/min-staff`

获取各日期类型最低排班人数配置。

**响应示例：**
```json
{
  "普通工作日": 2,
  "假前一日": 2,
  "周末": 1,
  "调休工作日": 1,
  "法定节假日": 1
}
```

#### `PUT /api/config/min-staff`

保存最低人数配置。

---

### 5.3 成员法定节假日排班天数

#### `GET /api/config/member-holiday?month=yyyy-MM`

获取指定月份各成员的法定节假日排班天数（覆盖值）及该月法定节假日总数。

**响应示例：**
```json
{
  "data": { "彭小容": 2, "何咏琪": 1 },
  "legalHolidayCount": 3
}
```

#### `PUT /api/config/member-holiday`

设置某成员在某月的法定节假日排班天数（0 ~ 该月法定节假日总数）。

**请求体：**
```json
{ "month": "2026-04", "name": "彭小容", "days": 2 }
```

---

### 5.4 统计数据

#### `GET /api/stats`

获取当月日历统计数据，包含每日最低人数和实际排班人数、告警列表（意愿采集超标）。

## 六、核心数据结构（前端）

### 6.1 类型定义

```ts
// 节假日配置行
type HolidayRow = {
  date: string;          // "yyyy/MM/dd"
  name: string;          // 节日名称，如 "清明节"
  type: '节假日' | '调休上班日';
  isLegal: boolean;      // 是否法定节假日（影响排班计算）
};

// 成员
type MemberItem = {
  name: string;
  role: '租号客服' | '卖号客服' | '管理员';
  status: '在职' | '离职';
  inSchedule: boolean;   // 是否参与排班计算
};

// 已保存排班方案
type SavedSchedule = {
  id: string;
  planName: string;
  month: string;         // "yyyy-MM"
  schedule: Record<string, Record<number, '早' | '晚' | '休' | '假'>>;
  //                  ↑ 成员姓名      ↑ 日期（1-31）  ↑ 班次
};

// 班次对调请求
type SwapRequest = {
  id: string;
  fromUser: string;      // 发起人
  toUser: string;        // 接收人
  month: string;         // "yyyy-MM"
  swaps: SwapPair[];     // 对调的班次对
  status: '待处理' | '已同意' | '已拒绝';
  createdAt: string;     // ISO 时间字符串
};

type SwapPair = {
  fromDay: number;       // 发起人的日期
  fromShift: '早' | '晚' | '休' | '假';
  toDay: number;         // 接收人的日期
  toShift: '早' | '晚' | '休' | '假';
};

// 排班调整日志
type BoardEditLog = {
  id: string;
  month: string;
  editor: string;
  timestamp: string;     // ISO
  type: 'edit' | 'apply_new' | 'apply_overwrite' | 'swap_request' | 'swap_approve' | 'swap_reject';
  changes?: { name: string; day: number; fromShift: string; toShift: string }[];
  planName?: string;
  swapFrom?: string;
  swapTo?: string;
  swapPairs?: { fromDay: number; fromShift: string; toDay: number; toShift: string }[];
};
```

### 6.2 意愿数据的 key 格式

员工预填意愿存储为：
```ts
// submittedLeaveData 外层 key = "yyyy-MM"（月份）
// 内层 key = "yyyy-MM-d"（注意：日期不补零，如 "2026-04-5"）
Record<string, Record<string, '轮休' | '请假'>>
```

## 七、功能模块详解

### 7.1 排班看板（Tab: `calendar`）

**对应视图：** 主页默认打开的第一个标签页。

**核心功能：**

| 功能 | 描述 |
|---|---|
| 日历视图 | 以月历方式展示全员排班，区分早班/晚班/休/假，颜色区分节假日类型 |
| 表格视图 | 以横轴为日期、纵轴为人员的二维表展示排班 |
| 月份切换 | 在当月/下月之间切换查看（左右箭头导航） |
| 仅看自己 | 勾选后日历只显示当前用户排班，过滤其他人 |
| 编辑排班 | 管理员可进入编辑模式，点击单元格修改班次，保存后记录调整日志 |
| 班次对调 | 员工可发起换班申请，选择对调对象及日期后提交，等待对方确认 |
| 调整日志 | 面板显示本月所有排班变更记录（编辑、应用、对调审批） |

**编辑模式说明：**
- 点击「编辑排班」进入编辑模式，此时表格/日历所有单元格可点击；
- 点击单元格弹出下拉菜单，可切换为「早/晚/休/假」；
- 点击「保存」弹出二次确认弹窗，确认后写入 `appliedSchedules` 并追加日志；
- 点击「退出编辑」放弃所有草稿，恢复到原始状态。

**班次对调流程：**
1. 点击「发起对调」，选择对调对象（其他成员）；
2. 选择自己的日期+班次 与 对方的日期+班次（支持多组对调）；
3. 系统自动校验合规性（早晚双向对换才合法）；
4. 提交后，对方在「待处理对调」区域可看到申请，选择同意/拒绝；
5. 同意后系统自动更新 `appliedSchedules` 并记录日志。

**调整日志事件类型：**

| 事件类型 | 触发时机 |
|---|---|
| `edit` | 进入编辑模式保存班次改动 |
| `apply_new` | 首次应用排班方案到某月 |
| `apply_overwrite` | 覆盖已有方案应用 |
| `swap_request` | 发起班次对调申请 |
| `swap_approve` | 对调申请被同意 |
| `swap_reject` | 对调申请被拒绝 |

### 7.2 休息日预填（Tab: `leave-request`）

**使用对象：** 全员（每位客服在排班计算前填报下月意愿）

**核心功能：**

| 功能 | 描述 |
|---|---|
| 月份选择 | 默认为「当月+1」（次月），可前后切换 |
| 日历点选 | 点击日历中的日期，弹出「轮休/请假/取消」三选一 |
| 提交意愿 | 点击「提交」将草稿写入 `submittedLeaveData`，同月再次提交可覆盖 |
| 连班校验 | 提交时检查：去掉轮休/请假日后，连续工作日不得超过 6 天，否则提示错误 |
| 已提交状态 | 已提交的月份在统计区显示轮休天数/请假天数 |
| 开放预填开关 | 管理员可在排班计算页面控制是否「开放预填」，关闭后普通员工不可修改 |

**视觉规则：**
- 🟢 绿色圆点 = 轮休
- 🔴 红色圆点 = 请假
- 普通工作日 = 白色格子
- 周末 = 黄色背景
- 节假日 = 绿色背景
- 调休工作日 = 紫色背景

### 7.3 排班计算（Tab: `schedule-calc`）

> 此模块为**管理员专属**，普通员工不可见。

**功能区域：**

#### ① 月份 & 意愿汇总区

- 管理员选择计算月份（默认下月）；
- 展示当月各成员已提交意愿天数（轮休 + 请假）；
- 显示「开放预填」开关：关闭后员工无法修改填报；
- 成员法定节假日排班天数（每人可单独覆盖，默认取全局法定节假日数）。

#### ② 排班日历预览区

在计算前可查看当月排班日型，每天显示：
- 日期类型（工作日/周末/节假日/调休工作日/假前一日）；
- 最低排班人数（可按日单独覆盖）；
- 节假日名称（如「清明节」「劳动节」）。

管理员可进入「编辑日历」模式，对单日进行日期类型和最低人数的精细调整。

#### ③ 开始排班计算

点击「开始排班计算」后系统弹出计算进度弹窗，实时展示日志。计算完成后（或重试完成后）弹窗下方出现：
- 「重新排班」：清空结果重新计算；
- 「保存排班」：输入方案名称后保存到 `savedSchedules`。

#### ④ 已保存方案列表

- 展示所有已保存方案，显示月份、方案名、保存时间；
- 每个方案可「查看/预览」「编辑」「应用到排班看板」「删除」；
- 应用时若当月已有方案，弹出覆盖确认弹窗；
- 应用成功后弹出成功提示，提供「前往看板」快捷跳转。

### 7.5 全局配置（Tab: `settings`）

> 此模块为**管理员专属**，普通员工不可见。

#### 节假日配置

- 以表格展示所有配置的节假日/调休工作日：日期、节日名称、类型、是否法定；
- 支持批量新增：点击「批量新增」弹窗，粘贴以下格式文本一键导入：
  ```
  2026/10/01 国庆节 节假日 true
  2026/10/07 国庆节 调休上班日 false
  ```
- 数据保存到 `/api/config/holidays`（Prisma AuditLog）；
- 删除单条记录，可实时预览变更后保存。

**字段说明：**

| 字段 | 값 | 说明 |
|---|---|---|
| type | `节假日` | 当天休息，不排班 |
| type | `调休上班日` | 周末但需要正常上班（视为工作日计入排班） |
| isLegal | `true` | 计入员工「法定节假日排班」天数指标 |
| isLegal | `false` | 不计入法定节假日统计 |

#### 最低排班人数配置

按日期类型设定每日最低上班人数，保存到 `/api/config/min-staff`。

| 日期类型 | 默认值 |
|---|---|
| 普通工作日 | 2 |
| 假前一日 | 2 |
| 周末 | 1 |
| 调休工作日 | 1 |
| 法定节假日 | 1 |

## 八、排班算法说明

排班计算在 `performScheduleCalc()` 函数（`page.tsx` 约 1000–1400 行）中执行，分为以下步骤：

### 步骤 ① 节假日解析

从 `globalHolidayConfig` 读取当月节假日，分类为：
- `法定节假日`（节假日类型）
- `调休工作日`（调休上班日类型）
- 普通工作日、假前一日、周末

计算：
- **工作日总天数**（包含假前一日和调休工作日）
- **法定节假日总数**

### 步骤 ② 意愿解析

从 `submittedLeaveData`（当前用户）和 `otherTeamsLeaveData`（其他成员）读取：
- `restWish[人名]`：Set<日期天数>，该人希望轮休的日期
- `leaveWish[人名]`：Set<日期天数>，该人希望请假的日期

### 步骤 ③ 个人额度计算

为每位参与排班的成员计算：

```
请假天数(leaveCnt)       = leaveWish[人名].size
应排班天数(requiredWorkDays) = 普通工作日天数 + 法定节假日排班数 - 请假天数
应休息天数(requiredRestDays) = 当月总天数 - 应排班天数（含假）
纯轮休天数(requiredOffDays)  = 应休息天数 - 请假天数
```

### 步骤 ④ 轮休日分配

- 优先将意愿轮休日分配为「休」；
- 若意愿日过多，随机选取满足额度的天数；
- 若意愿日不足，从非法定节假日、非假前一日的日期中补充随机「休」；
- 请假意愿日直接标记为「假」。

### 步骤 ⑤ 早晚班分配

- 以工作日池中排除「休」「假」后的天数为目标；
- 计算每日早晚班目标人数（按全员均分）；
- 强制规则：前一天排晚班的成员，当天必须继续排晚班（防止连续跨段）；
- 使用轮转调度确保每人早晚班均衡。

### 步骤 ⑥ 法定节假日处理

- 按 `memberHolidayDays` 中各成员应排法定节假日天数；
- 在法定节假日中为各成员分配「早」或「晚」（其余为「休」）。

### 步骤 ⑦ 日志输出（早晚班约束校验）

记录每日是否触发「强制晚班」规则的日志。

### 步骤 ⑧ 最低人数校验与强制补班

遍历所有工作日，若当日实际上班人数 < 最低要求，则将最早的轮休人员强制改为「早班」。

### 步骤 ⑨ 结果验证输出

对每位成员输出验证摘要（打印到计算日志）：

```
彭小容：上班 ✅22/22 （早11+晚11） | 轮休 ✅8/8 | 请假 0 | 法定节假日排班 ✅1/1，轮休满足率87.50%
```

- **轮休满足率** = 意愿轮休/请假日中实际被排为休/假的天数 / 总意愿天数 × 100%

若验证不通过（某人实际上班天数 ≠ 应排天数），返回 `null` 触发自动重试。

### 步骤 ⑩ 逐日明细输出

打印逐日早班人员、晚班人员、休假人员列表。

### 重试机制

- 最大重试次数：**8 次**；
- 每次失败等待 50ms 后重试（随机性保证每次结果不同）；
- 8 次全部失败后输出错误提示，建议检查配置。

## 十、数据持久化策略

系统采用**双轨持久化**：

### 10.1 localStorage（前端持久化）

| localStorage key | 存储内容 | 类型 |
|---|---|---|
| `cs_members` | 成员列表 | `MemberItem[]` |
| `cs_submitted_leaves` | 用户已提交的轮休/请假意愿 | `Record<string, Record<string, '轮休'\|'请假'>>` |
| `cs_saved_schedules` | 所有已保存排班方案 | `SavedSchedule[]` |
| `cs_applied_schedules` | 各月已应用排班方案 | `Record<string, SavedSchedule>` |
| `cs_swap_requests` | 班次对调申请列表 | `SwapRequest[]` |
| `cs_board_edit_logs` | 排班调整日志 | `BoardEditLog[]` |
| `cs_prefill_open` | 各月预填开放状态 | `Record<string, boolean>` |
| `cs_day_min_staff_overrides` | 单日最低人数覆盖 | `Record<string, number>` |
| `cs_swap_v2` | 对调数据版本标记（防旧格式污染） | `"1"` |

### 10.2 数据库（Prisma + SQLite）

| 配置类型 | API 路径 | AuditLog action |
|---|---|---|
| 节假日配置 | `/api/config/holidays` | `holiday-config` |
| 最低排班人数 | `/api/config/min-staff` | `min-staff-config` |
| 成员法定节假日天数 | `/api/config/member-holiday` | `MEMBER_HOLIDAY_OVERRIDE:yyyy-MM` |

## 十一、使用说明（操作手册）

### 11.1 本地启动

```bash
# 安装依赖
npm install

# 初始化数据库
npx prisma generate
npx prisma db push

# 启动开发服务器（端口 3001）
npm run dev
```

访问 http://localhost:3001

### 11.2 普通员工操作流程

```
每月 25 日前完成意愿填报
       │
       ▼
1. 打开系统 → 点击「休息日预填」
2. 确认月份为「下月」（如 4 月）
3. 点击日历中希望轮休的日期 → 选「轮休」
4. 点击日历中需要请假的日期 → 选「请假」
5. 检查连班不超过 6 天
6. 点击「提交确认发布」
       │
       ▼
等待管理员出排班结果后，前往「排班看板」查看
       │
       ▼
如需换班：点击「发起对调」→ 选择对象 → 选择日期班次 → 提交
```

### 11.3 管理员操作流程

```
每月 26 日后发起排班计算
       │
       ▼
1. 点击「排班计算」页签
2. 确认计算月份（默认下月）
3. 检查各成员意愿天数是否合理
4. 确认节假日配置正确（如有新假期前往「全局配置」更新）
5. 确认最低排班人数配置符合业务要求
6. 如有特殊成员节假日排班安排，修改「法定节假日排班天数」
7. 点击「开始排班计算」
       │
       ▼
等待计算完成（1-30秒，最多 8 次重试）
       │
       ├─ 计算成功 → 查看日志 → 点击「保存排班」→ 输入方案名
       │
       └─ 计算失败 → 检查配置（人数不足、意愿冲突等）→ 重新计算
       │
       ▼
在「已保存方案」中找到方案 → 点击「应用」
       │
       ▼
前往「排班看板」确认结果 → 通知员工
```

### 11.4 节假日配置更新

每年国家发布下一年节假日安排后（通常 11-12 月）：

1. 进入「全局配置」→「节假日配置」；
2. 点击「批量新增」；
3. 粘贴如下格式文本（每行一条）：
   ```
   2027/01/01 元旦 节假日 true
   2027/01/02 元旦 节假日 false
   2027/01/03 元旦 调休上班日 false
   ```
   - 字段顺序：`日期 节日名称 类型 是否法定`
   - `类型` 取值：`节假日` 或 `调休上班日`
   - `是否法定` 取值：`true` 或 `false`
4. 预览解析结果无误后点击「确认新增」；
5. 点击「保存配置」完成保存。

### 11.5 班次对调操作

**发起方：**
1. 在「排班看板」点击「发起对调」；
2. 选择对调对象（仅显示有当月排班的成员）；
3. 点击「新增对调组」，分别选择：
   - 自己的日期 + 想换出的班次；
   - 对方的日期 + 想换入的班次；
4. 确认后提交（系统校验：早↔晚 或 休↔工作班 的交换合规性）。

**接收方：**
1. 在「排班看板」看到「待处理对调」区域出现申请；
2. 点击查看详情，确认对调内容；
3. 选择「同意」或「拒绝」；
4. 同意后排班自动更新，并记录日志。

## 十二、已知限制与待办事项

### 12.1 当前已知限制

| 限制 | 说明 |
|---|---|
| 无真实登录鉴权 | `currentUser` 硬编码，所有人共享浏览器数据 |
| localStorage 单用户 | 意愿数据存在发起操作的浏览器本地，多人使用需在单一设备或共享方案 |
| 其他成员意愿为 mock 数据 | `otherTeamsLeaveData` 硬编码，需接入真实数据库读取 |
| 排班算法非生产级 | 当前算法已覆盖主要约束，极端情况（极少人数、超多节假日）可能重试失败 |
| 飞书通知未接入 | `feishu.ts` 目前仅打印日志，未发送真实消息 |
| 数据库仅用于配置 | 排班结果、意愿、日志均存在 localStorage，未持久化到数据库 |
